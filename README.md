# @tr-io/harness

Agentic development harness for Claude Code projects. Bootstraps and maintains the infrastructure that makes AI coding agents reliable, incremental, and self-correcting.

## What it does

`harness` scaffolds a structured set of files into any software project that:

- **Orients the agent** — CLAUDE.md table of contents, `.ai/` navigation maps, and session lifecycle protocol
- **Enforces quality via backpressure** — hooks that block pushes on lint/type/test failures; warnings for stale docs and bad branch names
- **Keeps context token-efficient** — progressive disclosure (short TOC → deep docs on demand), on-demand Linear ticket injection
- **Automates the ticket loop** — `harness auto` handles the full cycle from Linear ticket to merged PR

Built from [Anthropic](https://www.anthropic.com/engineering/harness-design-long-running-apps), [OpenAI](https://openai.com/index/harness-engineering/), and community harness engineering research.

## Installation

```bash
npm install -g https://raw.githubusercontent.com/tr-io/agentic-harness/main/tr-io-harness-0.1.1.tgz
```

Requires Node.js 20+.

> **Note:** Install via the packed `.tgz` rather than a git URL. npm always runs an inner `npm install --force` on git packages which strips the built `dist/` files.

## Quick start

```bash
# In a new or existing project directory
harness init

# Check harness health any time
harness check

# After upgrading the package
harness upgrade
```

## Commands

### `harness init`

Bootstraps the harness into the current directory. Works on both new and existing projects.

```bash
harness init              # Interactive — asks questions
harness init --no-interactive  # Use detected defaults without prompting
harness init --dry-run    # Preview files without writing
```

**What it does:**
1. Detects your stack (language, framework, linter, test framework)
2. Prompts for project name, commands, and Linear integration
3. Scaffolds all files based on your answers and feature toggles
4. For existing projects: runs a Sonnet sub-agent to generate filled-in codebase navigation maps
5. Merges into existing `.claude/settings.json` without losing your config
6. Scaffolds a GitHub Actions CI workflow if none exists
7. Suggests GitHub branch protection rules to set up manually

---

### `harness check`

Validates harness health. Reports pass/warn/fail per check.

```bash
harness check
```

Checks:
- `.harness.json` exists and is valid
- All mandatory artifacts present (CLAUDE.md, `.ai/` structure, hooks)
- Recommended artifacts present when features are enabled
- CLAUDE.md stays under ~100 lines
- Hook scripts referenced in `.claude/settings.json` exist on disk
- Commands configured in `.harness.json`

Exits with code 1 if any checks fail.

---

### `harness upgrade`

Upgrades harness templates after the package updates.

```bash
harness upgrade           # Interactive merge/overwrite/keep prompts
harness upgrade --dry-run # Preview changes
```

**Guarantees:**
- Additive-only — never changes existing defaults or removes features
- Files you've customized get a diff and a prompt (merge / overwrite / keep)
- Untouched scaffolded files update silently
- New feature toggles added as `false` by default

---

### `harness lint-setup`

Detects your stack and bootstraps a linter config if none exists.

```bash
harness lint-setup
```

| Stack | Linter | Formatter |
|-------|--------|-----------|
| TypeScript/JavaScript | Biome | Biome |
| Python | Ruff | Ruff |
| Rust | Clippy | rustfmt |
| Go | golangci-lint | gofmt |
| Java | Checkstyle | — |
| Ruby | RuboCop | RuboCop |

Never overrides existing linter configs.

---

### `harness auto <ticket-id>`

Fully automated ticket implementation loop. Requires `features.autoLoop: true` in `.harness.json` and a Linear API key.

```bash
harness auto TRI-42
harness auto TRI-42 --simplify   # Run /simplify on substantially changed files
```

**Flow:**
1. Fetch ticket from Linear
2. Assess complexity — if too large, propose splitting into sub-tickets
3. Spawn a new Claude Code session with ticket context and session protocol
4. Agent implements, tests, commits, and creates PR
5. Monitor CI — auto-fix failures (max 3 attempts, then escalates to you)
6. Wait for your PR approval and merge
7. Update ticket → Done

```bash
# Required environment variable
export HARNESS_LINEAR_API_KEY=lin_api_...
# or
export LINEAR_API_KEY=lin_api_...
```

## Configuration: `.harness.json`

Generated by `harness init`. Reference the JSON schema for editor autocompletion.

```jsonc
{
  "$schema": "https://raw.githubusercontent.com/tr-io/agentic-harness/main/schema.json",
  "version": "0.1.0",
  "project": {
    "name": "my-project",
    "type": "web-app",           // web-app | cli | library | monorepo | mobile
    "stacks": ["typescript"],
    "testCommand": "npm test",
    "lintCommand": "npx biome check .",
    "typeCheckCommand": "npx tsc --noEmit",
    "buildCommand": "npm run build"
  },
  "linear": {
    "enabled": true,
    "teamKey": "TRI"             // Your Linear team key
  },
  "features": {
    // Recommended — on by default
    "adr": true,                       // .ai/adr/ architecture decision records
    "testingDocs": true,               // .ai/testing/ testing conventions
    "completionReminder": true,        // Stop hook: self-review checklist
    "branchNamingWarning": true,       // Warning on non-conforming branch names
    "linterBootstrap": true,           // Bootstrap linter if none detected
    "linearIntegration": true,         // Inject ticket context into sessions
    "artifactFreshnessCheck": true,    // Warn on stale .ai/ docs after commits

    // Optional — off by default
    "dddContextMaps": false,           // .ai/ddd/ domain-driven design maps
    "latMd": false,                    // lat.md knowledge graph integration
    "evaluatorQA": false,              // Separate evaluator agent pass
    "autoLoop": false,                 // harness auto command
    "keelEnforcement": false           // keel structural code enforcement
  },
  "hooks": {
    "prePush": {
      "lint": true,
      "typeCheck": true,
      "unitTest": true
    }
  }
}
```

## Scaffolded structure

```
project-root/
├── .harness.json                          # Harness config
├── CLAUDE.md                              # Short TOC (~100 lines) — always loaded
├── .claude/
│   ├── settings.json                      # Hook configuration
│   └── hooks/
│       ├── pre-push-check.js              # Mandatory: blocks push on failures
│       ├── branch-naming-warn.js          # Recommended: branch name warning
│       ├── completion-reminder.js         # Recommended: self-review on Stop
│       └── artifact-freshness.js          # Recommended: stale doc warning
├── .ai/
│   ├── README.md                          # Doc index
│   ├── agent-instructions/
│   │   ├── session-protocol.md            # 6-step session lifecycle
│   │   ├── pre-plan.md                    # Branch creation + ticket lookup
│   │   └── pre-push.md                    # Pre-push checklist
│   ├── codebase/                          # Navigation maps (fill in or sub-agent generates)
│   ├── adr/                               # Architecture decision records
│   ├── testing/                           # Testing conventions
│   └── manifest.json                      # Source path → doc mappings
└── .github/
    └── workflows/ci.yml                   # Stack-aware CI workflow
```

## Hook behavior

| Hook | Trigger | Blocks? | Purpose |
|------|---------|---------|---------|
| `pre-push-check.js` | `git push` | **Yes** | Lint + type check + unit tests |
| `branch-naming-warn.js` | `git checkout -b` | No | Warn on non-`ticket-id-description` branches |
| `completion-reminder.js` | Session end | No | Self-review checklist + /simplify offer |
| `artifact-freshness.js` | `git commit` | No | Warn when .ai/ docs may be stale |

The pre-push hook reads commands from `.harness.json` and fails with actionable messages that tell the agent exactly what to fix. Individual checks can be disabled:

```json
"hooks": {
  "prePush": { "lint": true, "typeCheck": false, "unitTest": true }
}
```

## Linear integration

When `features.linearIntegration: true` (default) and `linear.enabled: true`:

- **Ticket context injection**: when working on a branch named `tri-42-add-auth`, the session protocol instructs the agent to fetch TRI-42's title, description, and acceptance criteria from Linear — on demand, not preloaded
- **Status updates**: ticket moves In Progress → In Review → Done at the right lifecycle points
- **PR linking**: PR descriptions auto-include a link to the Linear ticket

Requires the Linear GitHub integration for auto-linking: https://linear.app/settings/integrations/github

## Session protocol

Every Claude Code session in a harnessed project follows this lifecycle (defined in `.ai/agent-instructions/session-protocol.md`):

```
1. ORIENT       — read CLAUDE.md, git log, fetch ticket if applicable
2. VERIFY       — run tests + lint to confirm baseline is clean
3. PLAN         — identify ONE task, create branch, outline approach
4. IMPLEMENT    — incremental commits, conventional format
5. TEST         — unit tests + lint for changed code
6. FINALIZE     — self-review checklist, update .ai/ docs, push, create PR
```

One task per session. When a ticket is too large, the agent is instructed to propose splitting before implementing.

## Recommended GitHub setup

`harness init` will suggest these — configure them manually in your repo settings:

- **Require PR reviews before merging** (Settings → Branches → Branch protection)
- **Require status checks to pass** (select your CI workflow)
- **Disable force push to main**
- **Install Linear GitHub app** for ticket ↔ PR auto-linking
